For Unisys, providing universal access to the application is long-established institutional policy, and also is required by state or federal regulations. It is critical, therefore, that you validate your designs and page templates and the content throughout the development process to ensure that all pages are accessible to all users. Use the guidelines and techniques developed and maintained by the Web Accessibility Initiative (WAI) as a measure against which to test the accessibility of pages.

This is a reference document for accessibility principles and design ideas.

1.0 Images, Structure, Functions and Navigation

1.1 U-LEAF content MUST make sense without reference to images UNLESS the subject matter can only be displayed via images (e.g. an 'icon').

1.2 You MUST provide a text equivalent for every non-text element on your HTML pages. (Source: W3C, Web Content Accessibility Guidelines(WCAG) 2.1). In certain circumstances, the content of this text equivalent MAY be null.

1.3 Elements of HTML pages such as <img> MUST include an alt attribute.

1.4 Notes on Image usage

  • 1.4.1 Images that are purely presentational and are not content SHOULD be in the CSS, and SHOULD ONLY be in the HTML of the page. (Example: Buttons – Save, Add, Delete)
  • 1.4.2 Images that are, however, in the HTML SHOULD follow the alt attribute rules defined in section 1.3 above.
  • 1.4.3 You SHOULD ONLY use transparent spacer images, as their function can be provided by CSS layout.

1.5. Graphic Image

  • 1.5.1 : You SHOULD ONLY render text in a graphic image where the text is part of the brand image or Where characters/symbols cannot be assumed to be in the browser's standard character set (for example, mathematical/scientific symbols);

    • 2.0 Use of colour

    2.1 Introduction

    2.1.1 These standards ensure that the use of colour on U-LEAF is suitable for people who have some form of colour-blindness, vision impairment or have difficulty in reading text.

    2.2 colour contrast for text and images of text

    2.2.1 All colour combinations MUST pass the WCAG 2.1 AA-compliant colour contrast check in accordance with the Snook colour contrast checker. This means you need a 'YES' result against:

    • 'WCAG 2 AA Compliant' for normal text smaller than 18 pt; and/or
    • 'WCAG 2 AA Compliant (18pt+)' for normal text larger than 18 pt or bold text that is 14 pt or larger.

    N.B. The 'YES' result overrides any other issues that may be indicated such as Brightness or Colour Difference

    2.2.2 This requirement applies to text and images of text (not including incidental text, e.g. text on a street sign that happens to appear within a photograph). Stylised text used in logos and branding is exempt from this requirement.

    To test whether color contrast is sufficient to be read by users with color deficiencies or by those with low resolution monitors, print pages on a black and white printer (with backgrounds and colors appearing in grayscale). Also try taking the printout and copying it for two or three generations to see how it degrades. This will show you where you need to add redundant cues (example: hyperlinks are usually underlined on Web pages), or whether the cues are too small or indistinct to hold up well.

    2.3 colour blindness

    2.3.1 You MUST NOT use any colour combinations which will be indistinguishable by people with a form of colour-blindness (see Colour Vision Simulator Examples for details).

    2.3.2 You MAY use Colour-blindness image checker on your Image Files.

    Instructions:

    • Take a screen grab (PC) or use Paperazzi (Mac) to grab an image of your page.
    • On the Color Blindness site Drag and drop or paste your file in the section as in screenshot
    • Select Options based on view needed.
    • Check that the colour blindness emulation complies with the Colour Contrast Standard.
    • Repeat the above for each type of colour blindness views: Trichromatic, Dichromatic and Monochromatic.

    2.3.5 You SHOULD also check that your colour choices will not interact with each other when they are displayed on a CRT/TFT (gamma, moiré, bleeding etc.).

    2.4. Use of CSS

    2.4.1 You MUST define both the text and background colours of all textual areas on a page in a CSS style sheet, so users can override your colours with a style sheet of their own.

    For further information, see the CSS standards and HTML Integrity Standards.

    3.0 Text Equivalent

    3.1 Alt attribute

    3.1.1 Alt attributes provide a text description of images. They SHOULD be:

    3.1.1.1 Concise: Unnecessarily long alt attributes are time consuming.

    3.1.1.2 Meaningful: Non-specific text does not provide sufficient information to users who cannot see the image.

    3.1.1.3. Useful: They SHOULD describe the functional intent of the image.

    3.1.1.4 Presentational images in a page's HTML MUST carry an alt attribute which MUST be blank (that is, alt=""). Providing a blank alt attribute will allow screen-readers to skip the image.

    3.1.1.5 Graphic Text: The alt attribute of an image that features functionally significant text MUST include that text. If you require a call to action this, this can be achieved by having a text link placed over the picture.

    3.1.1.6 Functional images (for example, graphical buttons): If an image is essentially functional (for example, the "Save" or "Delete" button) the alt attribute MUST convey the function, or include the exact text featured on the button. The alt attribute MUST NOT carry a description of what the button looks like.

    3.1.1.7 Image maps: With image maps you MUST provide an alt attribute to every tag.

    3.1.2 Words you MUST NOT use in the alt attribute

    3.1.2.1 Screen readers tell the user that an image is an image; therefore, there is no need to duplicate this information in the alt attribute.

    3.1.2.2 You MUST NOT use phrases such as "image of", "picture of", "photo of" unless that fact has functional significance.

    3.1.2.3 You MUST NOTE use the words "brand" or "logo" in an alt attribute unless the fact that it is a logo or brand is editorially significant.

    3.2 Title Attribute

    3.2.1 Title attributes MUST NOTE be used UNLESS the wording of the link text does not inform the user of the destination of the link, when read in isolation.

    3.2.3 You MUST NOTE simply repeat the link text in the title attribute.

    3.2.4 Title attributes MUST be used to describe or support the description of a link's destination; whether this link is around an image or text. This includes tags for image maps.

    3.2.5 Title attributes MUST contain human-readable data

    3.2.6 The title attribute MUST NOTE replicate the content of the alt attribute, but MAY include some of the same wording.

    3.2.7 Title attributes should be:

    3.2.7.1 Useful: They SHOULD ONLY be used where the destination description is not already provided in a text form, UNLESS they are supporting that definition with additional information.

    3.2.7.2 Concise: Unnecessarily long title attributes might not be displayed in the best possible way when appearing as "tooltips" as a user moves their mouse over a titled link. Title attributes SHOULD ONLY be more than 30 characters long.

    3.2.8 You SHOULD ONLY use the following words in a title tag:

    3.2.8.1 "Link to", "jump to": you do not need to start your description with these phrases because users are normally already prompted with this information by their assistive technologies.

    3.2.8.2 "follow this link": this does not provide information about the link destination.

    3.2.8.2 "click here": this assumes users are using a mouse, which they may not be.

    3.2.8.3 "More...","More information..." screen readers strip the links from a page, which means they are often presented to the user out of context. Links MUST contain information about their destination.

    4.0 Text Link Standards

    4.1 Requirements

    4.1.1 Link text MUST describe the destination of the link.

    4.1.2 Title attributes SHOULD ONLY be used when the link's text does not fully describe the destination of that link.

    4.1.3 A text link's title-attribute SHOULD further describe the destination of the link and MUST NOT only repeat the link's text (refer to the Textual Equivalent Standards for more information).

    4.1.4 Example: If you roll over a button in Special Arrangement screen says "Delete". The title-attribute will say "Delete Special Arrangement"

    4.1.5 Link text MUST be meaningful out of context e.g. do not use "More", "Further Information", "Visit", "Watch Now", "Download" or "Click Here". You MAY hide further descriptive text in a manner that can be found by screen readers.

    4.1.6 You MUST NOT use the URL as link text, UNLESS displaying dynamic contents (such as search results) where there is no alternative.

    4.2 Making links selectable

    4.2.1 A text link SHOULD have sufficient space around it for the user to be able to accurately select the link, particularly where several links appear together.

    4.2.2 You SHOULD have a minimum of four pixels padding around a link; or equivalent, given your chosen units of measurement.

    4.3 Making text self-evident

    4.3.1 Links MUST be clearly identifiable and distinguishable from surrounding text, by use of a presentation or context cue.

    4.3.1.1 A presentation cue MUST be one or both of these:

    • bold and underlined
    • a different colour (all colours used MUST comply with colour contrast standards, including being visible to colour blind users).

    4.3.1.2 A context cue is when a link's position e.g. left-hand navigation or related links box conveys that it is clickable. In these cases use of bold is sufficient however visual elements SHOULD be used to indicate their significance, e.g. graphics or bullets.

    4.3.2 You MUST use a mouse-over state-change to indicate that a piece of text is a link. This MAY be either an underline or a change of colour. Underlines SHOULD ONLY disappear on mouse-over UNLESS the link is a graphic and has some other graphic form indicating a change of state.

    4.3.3 Your visited link colour SHOULD be different from the non-visited links' colour. This colour difference MUST be evident to colour blind users.

    5.0 Changing browser settings for accessibility testing

    To set Internet Explorer to ignore the colours and fonts used in web sites, first go to the 'Tools' menu and choose 'Internet Options'. Then click on the button, then tick the, 'ignore font styles' and 'ignore font sizes' boxes . Then click OK to apply the settings.

    To change just the size of text go to 'View - text size' and choose the setting you are most comfortable with (largest, larger, medium, smaller, smallest).

    You may also find it useful to change the default colours and fonts used on your computer. The procedure for doing this is different for different versions of Windows and other operating systems. You should therefore consult your computer's Help facility for instructions on doing this. Further information is available from AbilityNet's 'My computer my way' web site - see http://www.abilitynet.org.uk/myway.

    6.0 U-LEAF Browser Support Standards

    1.0 Introduction

    1.1 Scope of this Document

    1.1.1. This standard applies to web browsers that are being used on desktop versions of Windows, Mac OS and Linux/Unix. This does not cover Tablets, PDAs, mobiles, and other mobile devices.

    1.2. Web Browser Support

    1.2.1. Unisys accepts that the nature of the web medium is such that web pages cannot be produced in such a way as to be uniformly rendered in all browsers, so as to provide a consistent experience for all users. We accept that small variations in this experience are acceptable within the 'Levels of Support'.

    1.2.2. Web browsers are assigned a 'Level of Support' by the U-LEAF Browser Support Standards Work Group. These levels of support are summarised in the Browser Support Table. (See Browser Support Table for details of how these levels of support are decided).

    1.2.3. There's nothing wrong with using all the latest bells and whistles to support funky features of newer browsers, but try to do it in a way that still allows users not supporting (or intentionally disabling) these features to access your basic content.

    2.0 Level of Support

    2.1. Support Level 1 - supported web browser - Support definition

    2.1.1. All content MUST be readable and usable and all functionality MUST work.

    2.1.2. Variations to presentation of content MUST be minimised.

    2.1.3. Where CSS layout is used, the CSS MUST be rendered by supported web browsers, so that a fully-styled version of the page is presented to the user.

    2.1.4. Variations between browsers are inevitable. In these situations, when deciding which browsers should offer a better experience than other browsers (i.e. which would be the closest to the original design), you MUST base your decision on which outcome would maximise the 'Objectives of Web Browser Support'.

    2.1.5. Pages SHOULD be developed to maximise the user experience for users of the web browser with the highest proportion of users (IE7 PC at this time) UNLESS this would greatly compromise the 'Objectives of Web Browser Support'.

    2.2. Support Level 2 - partially-supported web browser - Support definition:

    2.2.1. All core content MUST be readable and usable and navigation MUST work.

    2.2.2. Any degradation to (client-side) application functionality MUST be graceful degradation.

    2.2.3. Any degradation to presentation MUST NOT obscure content.

    2.2.4. Where CSS layout is used, you MAY choose to provide an unstyled version of the page to partially-supported browser.

    2.3. Support Level 3 - unsupported web browser - Support definition:

    2.3.1. No support or testing necessary.

    2.3.2. Any web browsers not specifically listed in the support table MAY also be regarded as level 3; that is, unsupported.

    3. Objectives of Web Browser Support

    When considering which 'Level of Support' a web browser should be assigned, the following are some of the issues that are considered.

    1. U-LEAF should provide value for money to its users

    • Web application standards are good for the application and meeting these standards offers users value for money.
    • U-LEAF Users use a variety of web browsers. U-LEAF aims to provide the best possible experience to the largest number of people.
    • Some users do not have the choice of using a popular, modern web browser.
    • Some users use accessibility tools - these users benefit greatly from web pages that are standards compliant. U-LEAF should not, where possible, provide a service that gives a competitive advantage to any particular product or service. (e.g. we should not add to any page 'works best in x browser').

    7.0 Reliable Cross-Browser Testing tools

    Recommended IE Testing Solutions

    1. Virtual Machines + IECollection

    One of the best solution is using dedicated virtual machines (VMs) and a tool called IE Collection.

    Utilu's IE Collection offers the following:

  • A tested collection of standalone versions of IE, including versions 1 through 9;
  • Confirmed support for accurate rendering when compared against the original IE executables;
  • Confirmed support (to date) for the correct IE JavaScript engine implementations that shipped with the originals;
  • Confirmed support for the correct UA strings being detectable;
  • Support for both the 32-bit and 64-bit versions of Windows XP, 2003, Vista and 7;
  • Access to the IE Developer Toolbar, which comes with the standard IE Collection set-up; this is compatible with IE 5 and above, but you also have the option to install Firebug Lite if you prefer that.

    • Although the majority of IE versions are supported and function reliably within the collection, there are known issues with versions of IE 7 when used under Windows Vista or Windows 7 (as noted on Utilu's website)

    2. Dedicated VMs for Each Version of IE

    Developers might wish to maintain a dedicated VM for each version of IE being tested. On these VMs, rather than using any third-party tools, they would install the original, official IEs instead.

    The reasoning behind this is that it potentially minimizes the risk of individual browser configurations or overrides from interfering with one another, and it ensures that you get only the exact version of IE that you wish to test (and, in turn, the corresponding JavaScript engine in that version).

    If you're setting up dedicated VMs for individual releases of IE, I recommend the following set-up:

  • Windows XP (SP3) for IE 6,
  • Windows Vista (SP2) for IE 7,
  • Windows Vista (SP2) for IE 8,
  • Windows 7 for IE 9 and 10

    Proposed Accessibility Guidelines

    1.0 Keyboard Access

    1.1 Introduction

    Keyboard access to active elements of a page is important for many users who cannot use a pointing device (mouse) to navigate around the page. It is recommended for U-LEAF to reach the required accessibility standards for the future release.

    1.2 Tabbing

    1.2.1. All functionality on all U-LEAF applications and services SHOULD be fully controllable using the keyboard alone.

    1.2.2 Pages SHOULD provide a logical tab order through elements, form controls and objects, to increase the pages' ease of navigation via the keyboard.

    1.2.3 A logical tab order is one which makes sense when the layout is removed when vocalised by a screen reader – for example, elements which need to be read together (a heading and list of options) should be coded such that they are vocalised together rather than broken up by other elements.

    1.2.3.1. For HTML pages, you SHOULD do this in the way you code the page design in HTML.

    1.2.3.2 You SHOULD NOT use a positive tabindex value. You MAY use a tabindex of 0 or -1 under certain circumstances where this can increase the accessibility of your Javascript. For further clarification, see W3C HTML 4.0 recommendations and also draft W3C HTML 5.0 recommendations and W3C Aria recommendations.

    1.3 Keyboard Shortcuts

    1.3.1 You SHOULD include keyboard shortcuts to enable keyboard users to jump more quickly around the interface using the keyboard.
