This document describes the use of Cascading Style Sheet (CSS) on U-LEAF screens. CSS can be used to minimise the code required to render a site and using with care and particular attention for cross-browser compatibility, it could provide reduced download times and simplify application maintenance.

A full discussion of CSS rules is beyond the scope of this document. Further resources are listed in the References section. This document MAY be more understandable when read in conjunction with the Browser Support Standards and XHTML Standards.

1. General principles

1.1. CSS MUST be used to specify presentational aspects of page design such as typography, layout, interaction (along with JavaScript) and design features.

1.2. The page MUST be usable and readable without the CSS.

1.3. Style sheets SHOULD be tested against all level 1 and level 2 browsers, see Browser Support Standards.

  • 1.3.1. In Level 1 browsers the CSS MUST render a fully usable representation of the page design. In level 2 browsers a page MUST be readable and usable, but doesn't have to completely represent the design.

1.4. Use of CSS MUST be valid according to a published W3C recommendation, UNLESS you need to deviate from this to keep within the Unisys U-LEAF Browser Support Standards.

  • 1.4.1. If you wish to use browser specific syntax, you SHOULD do so in such a way as to limit the impact on future browsers.

1.5. The presentation of any page employing CSS MUST degrade gracefully (see: Browser Support Standards) in browsers that do not support, or provide only partial support for certain CSS features.

1.6. You MUST NOT use !important.

2. Templates

U-LEAF Stylesheets arranged

All Holmes Portal screens should have a similar ‘look and feel’. The base Liferay components and theme are provided to enable this. It is appreciated that the functionality within the various module portlets will differ significantly, however the standards detailed within this document aim to address all aspects of the user interface and ensure a consistent ‘look and feel’ at all times.

Styles used in U-LEAF

In general, all the styling of the Holmes Portal is achieved via the Holmes theme CSS files. The CSS files can be found at the following location:

holmes2ng_futures_SDS\holmes2_vob\holmes2ng_code\implementation\holmes2themes\Holmes-theme\src\main\webapp\css

It is important to point out that Liferay has a standard set of base CSS files and a specific one for customising.

Base CSS Files:

  • css > application.css This CSS includes all of the styling that is used for application elements, such as dialogs, inline popups, tabs, tags, and other elements.
  • css > aui.css: This CSS sets the default styles for the core, variables, mixins, path and other liferay specific styling components.
  • css > base.css: This CSS sets the default styles for the base components.
  • css > forms.css: This CSS sets the default styles for forms.
  • css > layout.css : This CSS controls the column layout of U-LEAF. It is not recommended that you change this file.
  • css > main.css : This is the main CSS that includes other CSS files.
  • css > navigation.css: This CSS sets the default styles for the main navigation.
  • css > portlet.css: This CSS sets the default styles for portlets.

Custom CSS File

  • css > custom.css: This is where you would put all of your new styles and all of your overrides of the styles in the parent theme. Custom.css is loaded last, and so styles in this file are able to override any styles in the parent theme. Best practice recommends that you make all your custom themes using only the custom.css file, and that you not override any of the templates unless absolutely necessary. This will make future upgrades far easier, as you won't have to manually modify your templates to add support for new Liferay features. Most CSS styles required for future development already exist within this CSS and are documented accordingly in the following sections for reference. Any subsequent additions to the custom.css should be approved via the development team leader

Deploying the Holmes Theme :

The Holmes theme has to be built and deployed for the CSS styles to take effect. The Holmes theme is a ‘project’ in its own right and needs to be built and deployed in a similar way to any other '.war files.' The .war file is named Holmes-theme.war and needs to be placed in the standard JBoss deploy directory on startup.

3. Implementing CSS

2.1. Definitions for this section:

  • External CSS files = using <link> in the <head>
  • Document head CSS = using <style> in the <head>
  • Inline CSS = using the style attribute on a tag, i.e. <div style="colour:red;">

2.2 External CSS files SHOULD be used in preference to document head CSS and document head CSS SHOULD be used in preference to inline CSS. There are several advantages to keeping all CSS rules in one or more separate files. The HTML documents become smaller, CSS files are stored in the browser cache and will only need to be downloaded once, and you only need to edit a single file to change the presentation of an entire website.

2.3. CSS files MUST have the file extension '.css' and SHOULD be stored in the relevant includes directory in the site structure, usually /style/.

2.4. The file size of CSS files SHOULD be kept as low as possible, especially on high demand pages.

2.5. External CSS MUST be linked to using the element which MUST be placed in the head section of the document.

2.6. External style sheets MUST NOT be imported (i.e. using @import) as it impairs caching

2.7. Document head CSS MAY be used where a style rule is only required for a specific page.

2.8. Inline styles MUST NOT be used.

2.9. Query string data (e.g. "style.css?v=0.1") MUST NOT be used on an external CSS file.

2.10. Use the right Doctype: The doctype declaration matters a whole lot on whether or not your markup and CSS will validate. In fact, the entire look and feel of your site can change greatly depending on the DOCTYPE that you declare.

2.11 Comment CSS: Just like any other language, it's a great idea to comment code in sections. To add a comment, simply add /* behind the comment, and */ to close it.

4. Typography

4.1. A single generic font-family name MUST be appended to the end of a font statement as a fall-back. This MUST be serif, sans-serif, cursive, or monospace.

For example: "helvetica, arial, sans-serif".

4.2. Typographic sizes MUST NOT be specified in units that are not resizable in all browsers such as px and pt, except for in print stylesheets.

4.3. Typography SHOULD be sized using em, but MAY be sized in % or keyword values.

4.4. Text MUST be readable when the text size is increased by two steps in any of the Level 1 supported browsers (Note: For browsers that support text resizing, as some only zoom the page).

4.5. Default U-LEAF font is Verdana

5. Links

5.1. Link properties SHOULD be set using CSS, including control of the underlining of links where appropriate.

4.2. When styling links the order is important, and as such you SHOULD order them using the LVHA method.

6. Page colours

6.1. Always ensure that the use of colour on ULEAF is suitable for users who have some form of colour-blindness, vision impairment or have difficulty in reading text.

6.2 You MUST always define a page background colour in your CSS, even if it is white.

6.3 When specifying a text colour or a background image, you MUST specify a background colour that compliments it and vice versa.

6.4 A full default set of definitions for colours (equivalents of text, link, alink, vlink and bgcolour in html) MUST be specified in the page's CSS style-sheet.

6.5 Please also refer to the Use of Colour and Colour Contrast Standard.
