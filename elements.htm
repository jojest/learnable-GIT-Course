U-LEAF Template Elements

This section defines the use of HTML elements on U-LEAF functional pages.

U-LEAF SMART & TAG Management interface elements are not covered in this styleguide.

1. Soft Entities

Soft Entities are a major feature introduced in the U-LEAF workstream that allows simple entity types to be created without the need for software changes. In place of the hard-coded approach to describing the characteristics and fields of an entity, a dynamic meta-data based approach is used. This meta-data is maintained in the U-LEAF database. The meta-data describes the type and format of the data fields of each soft entity, the rules for processing the data and simple field presentation information. Using this information, web pages are dynamically generated to perform entity searching, viewing, creation and update, and meta-data based validation rules are applied to each entity prior to storage.

For further technical details refer JSP Wiki > Soft Entities

2. Portlets

Portlets are pluggable user interface software components that are managed and displayed in U-LEAF. Portlets produce fragments of markup code that are aggregated into U-LEAf functional pages. Typically, following the desktop metaphor, U-LEAF functional page is displayed as a collection of non-overlapping portlet windows, where each portlet window displays a portlet. Hence a portlet (or collection of portlets) resembles U-LEAF that is hosted in Liferay portal.

All portlets should have the same portlet border and header style as shown below. This is provided as default by the standard portlet classes contained within the custom.css

For further information on Portlets, refer Liferay Website > Portlets section

The Portlet – Background Colour

The portlet background colour generally consists of one of two colours; white (#FFFFFF) or light grey (#F2F4F4)

The ‘smaller’ portlets often have the light grey background, whereas the larger transactional based portlets have white backgrounds with the content tables coloured accordingly.

It is important to point out that the standard default portlet background is defined as white in the custom.css, thus:


.portlet-content {
padding: 12px 10px 10px;
background:#FFF none;
}

This can be overridden by defining a specific CSS class for a portlet and defining as follows:


.portlet-incident .portlet-content {

background: #F2F4F4 none;

}

3. Capitalizing Titles, Sub Titles, Navigation, Form Labels and Text

Make titles, sub titles, labels and text for interface elements easy to understand and avoid technical jargon as much as possible. Try to be as specific as possible in any element that requires the user to make a choice, such as radio buttons, checkboxes, and buttons. Although it’s important to be concise, don’t sacrifice clarity for space.

All interface element Titles, Sub Titles and Labels should use Title style capitalization as follows

These are the words that should be capitalized:

  • - The first and last words of the title
  • - All nouns, pronouns, verbs, adverbs, and adjective
  • - Any conjunction or preposition of five letters or more --(This is optional. You can choose to go with an older rule that says not to capitalize prepositions or conjunctions regardless of length. The five letters or more standard has developed over the last two decades or so. Either form would be acceptable, as long as we use it consistently.)

These are the words that should not be capitalized:

  • - Articles (the, a, an), unless the article is the first or (less likely, of course) last word of the title.
  • - Prepositions of four letters or fewer (unless the preposition is the first or last word of the title). Prepositions convey the following relationships: agency (by); comparison (like, as . . . as); direction (to, toward, through); place (at, by, on); possession (of); purpose (for); source (from, out of); and time (at, before, on).
  • - Conjunctions of four letters or fewer (unless the conjunction is the first or last word of the title)
  • - The particle "to" used with an infinitive (unless the "to" is the first or last word of the title. For an example of the infinitive particle "to" capitalized at the end of the title, take my article "It's My Participle, and I'll Dangle It if I Want To". )

Text Style Guidelines

Text is prevalent throughout the ULEAF interface for such things as button names, menu labels, dialog messages, and help tags. Using text consistently and clearly is a critical component of UI design. First word text should be capitalized, and the rest of the words are lowercase, unless they are proper nouns or proper adjectives. If the text forms a complete sentence, end the sentence with proper punctuation; otherwise, do not add ending punctuation.

5. The Portlet – How to define a specific portlet CSS class to override the default CSS

All portlets have a liferay-portlet.xml.

The tag gives the ability to specify a CSS class which can be used to override default CSS classes within the custom.css. For example:

<?xml version="1.0"?>
<!DOCTYPE liferay-portlet-app PUBLIC "-//Liferay//DTD Portlet Application 5.2.0//EN" "http://www.liferay.com/dtd/liferay-portlet-app_5_2_0.dtd">

<liferay-portlet-app>
	<portlet>
		<portlet-name>softentity</portlet-name>
		<virtual-path>/holmes2/holmes2ng-softentity-ui</virtual-path>
		<instanceable>true</instanceable>
		<private-request-attributes>false</private-request-attributes>
		<private-session-attributes>false</private-session-attributes>
		<css-class-wrapper>portlet-softentity</css-class-wrapper>
	</portlet>
</liferay-portlet-app>

The CSS class defined as the <css-class-wrapper> is then used within the custom.css as follows, effectively overriding any default CSS styles for the default .portlet class :

.portlet-softentity .portlet {
	border: 1px solid #cad7d9;
	background: #FFFFFF none;
	margin-top: 20px;
	margin-left: 10px;
	margin-right: 24px;
	margin-bottom: 15px;
	padding: 0px;
	color: #1e4585;
}

    8. Status Messages - Success & Validation

    Currently U-LEAF supports Success and Validation messages to provide feedback for user actions.

    Web application design involves responding to errors in a way that is informative and helps users get back on track.

    Success message

    Success message(s) listed here

    Success messages should be displayed after user successfully completes an operation. For example, the message can say: "Entity profile has been saved successfully". This means that intended operation in this process (saving profile) has been successfully performed.

    Validation Message

    Please select a valid document type
     Please enter valid date taken
     Please enter a valid source reference

    When an error occurs, let the user know. Return the page with feedback right at the top of the section, indicating that an error occurred. An alert icon can be helpful in drawing attention to an error message, but be sure to include alternate text with the image (alt="Alert!") so that nonvisual users know that the icon and text comprise an error message. Provide specifics about what happened: "Your password is incorrect" rather than "Your username and password do not match." Provide this explanation next to the field in question—in this case, next to the password. Proximity counts! If the user makes an error, provide specific, descriptive help in the immediate area of the problem.

    In responding to errors, make sure error messages are tuned to users' needs. Error messages are too often written in programmer speak, either too detailed and specific or too vague, offering little in the way of explanation and guidance on where to go next ("Operation failed: try again"). Give users just enough information about what happened. They don't need to know that the error number is "404," but they do need to know that the page they requested cannot be found on the server. Provide guidance about what to do in response to the error.

    Note: Do not use field default text for essential instructions. Default text displays within an input field and can provide help and instructions without using precious screen space. However, default text disappears when the user activates the field to enter information, which is just when the instructions are needed most (Was that year, month, day, or day, month, year?). The best location for instructions is on the screen, alongside the corresponding form element.

    10 . ULEAF Wiki Help Configuration

    Technical information to create Wiki Help

  • (1) wiki-help.properties file location is under application server root folder i.e. under jboss-5.1.0.GA folder
  • (2) This properties file contains all the key-values where "key" holds the unique id of the Wiki Help page and "value" holds the actual Wiki Help page name. e.g: category.search=Category Search
      key : contains all lower case alphabets with dot in between e.g. category.search
      value : Wiki Help page name is Category Search
  • <3) NOTE: Make sure wiki-help.properties up-to-date before adding the new key-values in it
  • (4) In the jsp page just include the block as follows 
    <script>
  jQuery('.portlet-topper').prepend('<div id="wikiHelp" class="helpIcon" title="Help"></div>');
  jQuery('#wikiHelp').bind('click', function(){
	launchHelp('<h2ng:help pageNameKey="action.viewproofoflifeaction"/>');
  }); 
</script>
  • (5) Help icon(images/icons)/css(uleaf.css)/javascript(javascript.js) can be found under Holmes-theme project
  • (6) Help Tag library that reads the "wiki-help.properties" and brings the Page name for the given key can be found under common-presentation project

    Technical information to create Wiki Help for tabbed pages like SoftEntity

  • In the jsp page just include the block as follows 
    • <div id="helpLink" class="helpIconSoftEntity" title="add title" 
            onClick="launchHelp('<h2ng:help pageNameKey="${entity.specification.name}.view"/>')">
        </div>

    '${entity.specification.name}.view' is the page name key which will add the relevant page name. E.g nominal.view

    12. Titles

    You can add titles to essentially everything in HTML, including the entire range of form tags. Even if you're also using the accessible HTML catalogued in exacting detail here, there is no harm and possibly quite a bit of benefit in using titles in forms. There is some indication that certain screen readers have trouble wading through forms whose elements lack a title. That really means every single element – every check box, every radio button, every text-entry field, the lot. It is most useful to assign titles to input and textarea fields – anywhere you must enter your own information.

    Example

    :
    • HTML Markup
    • CSS

      HTML

      <label style="display:inline" for="namereq" title="Your full name">Name <strong title="Required" 
class="required">*</strong></label> : <input type="text" id="namereq" name="namereq" value="">

    14. Labels

    Labels may be hidden from the display by using display:none. This has the effect of not rendering the label on screen but, because it's in a form, screen readers will still read the label aloud. Note this technique only applies to form elements and that labels should be visible wherever possible.

    A form with text boxes and radio buttons that don't have labels (prompts or requests for information) is likely to be pretty well useless for U-LEAF application users.

    23. Data Tables

    Forms contain data and, if tabular, may be held in data tables. Data tables can be complex objects. Please ensure as a minimum:

    • A table summary is stated. Outlining the purpose of the tables contents .
    • Use a caption to state the purpose of the tables contents .
    • Define column and or row headers along with their scope
    • Define, where required, colgroups and their scope
    • Use and to separate content from headers where applicable .
    • Consider table linearisation carefully before separating a label from its associated input.

    Note: Usually constructs of form, legend, fieldset and label are sufficient to layout a form. Use a table only if the data input is tabular by nature. File Upload box and button:

    Remember when uploading files:

    • Do not allow spaces in the filename. Some browsers do not resolve spaces well. Use an underscore or a hyphen instead.

    4. Data Validations Error messages

    When an error occurs, let the user know. Return the page with feedback right at the top of the section, indicating that an error occurred. An alert icon can be helpful in drawing attention to an error message, but be sure to include alternate text with the image (alt="Alert!") so that nonvisual users know that the icon and text comprise an error message. Provide specifics about what happened: "Your password is incorrect" rather than "Your username and password do not match." Highlight the lable and field in question—in with red color. Proximity counts! If the user makes an error, provide specific, descriptive help in the immediate area of the problem.

    In responding to errors, make sure error messages are tuned to users' needs. Error messages are too often written in programmer speak, either too detailed and specific or too vague, offering little in the way of explanation and guidance on where to go next ("Operation failed: try again"). Give users just enough information about what happened. They don't need to know that the error number is "404," but they do need to know that the page they requested cannot be found on the server. Provide guidance about what to do in response to the error.
