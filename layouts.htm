The following document is a style guide to assist in the research and designing of U-LEAF application. To ensure design continuity and maintain the lifespan of application, it is important that everybody involved in the research, design maintenance and updating of sites read, understand and adhere to this guide. All U-LEAF Portal screens should have a similar ‘look and feel’. The base 'Liferay components' and 'theme' are provided to enable this. It is appreciated that the functionality within the various module portlets will differ significantly, however the standards detailed within this document aim to address all aspects of the user interface and ensure a consistent ‘look and feel’ at all times.

U-LEAF Legacy application layouts are not covered in this section.

1. Navigation system

The U-LEAF application navigation is one of the main ways to make information accessible for users. The navigation helps users to go through the information structure of the application and tells them where they are, where they have been, and where they can go.

This section outlines the main navigation elements:

  • The Primary Navigation as dropdown menu

This area must be consistent from page to page. It holds important navigational elements: quick link functions, and primary navigations with drop down menu options.

Drop down menu navigation section with a direct navigational link to home page. Items should accomodate the space of 1024px width in all circumstances. It is recommended to reduce the size of text if navigational link width exceeds specified width of 1024px. The location, width, height and appearance of this menu should not be changed from specified sizes.

Customization:

  • Size and location of this area is fixed.

Logo customization

  • U-LEAF logo could be replaced with client logo, but should be resized to accomodate the space avilable for logo. The logo is a graphic that links back to UFSS home page and flexible width and height of 29px. Margin around logo must be 10px for top and bottom, 20px for left and right. Logo must not exceed in height as the size specified which might cause the part of logo will not be visible. There must have margin defined around logo.
  • Background or logo must be transperant and recommended logo image format is PNG-24
  • The colour of the primary navigation blue area can be changed to accomodate the theme color customer is choosen.

6. Web Element Styling

In general, all the styling of the Holmes Portal is achieved via the Holmes theme CSS files. Styling of the portal container, navigation bar, portlet border and portlet header should be in the U-LEAF Theme project. Refer ULEAF Application Structure for further information on ULEAF THEME

It is important to point out that Liferay has a standard set of base CSS files and a specific one for customising. For example:

Base CSS Files:

application.css, base.css, forms.css, layout.css, main.css, navigation.css and portlet.css

The base CSS files, in general, should not be changed unless specified via the development team leader.

Custom CSS File:

custom.css

The custom CSS file is where specific styles relating to the ‘Holmes’ theme reside. These styles cover all aspects of portlet development ranging from the outer portlet wrapper, down to specific field based styling.

Most CSS styles required for future development already exist within this CSS and are documented accordingly in the following sections for reference. Any subsequent additions to the custom.css should be approved via the development team leader.

Deploying the Holmes Theme :

The Holmes theme has to be built and deployed for the SCSS(SASS)/CSS styles to take effect. The Holmes theme is a ‘project’ in its own right and needs to be built and deployed in a similar way to any other .war file. The .war file is named Holmes-theme.war and needs to be placed in the standard JBoss deploy directory on startup.

For further details on build and deployment of theme refer JSP WIKI - ULEAF2 Theme Development

The CSS files can be found at the following location:

uleaf-theme\docroot\_diffs\css

There are number of other possible locations for web element styling information in the application and CSS files and Images, should be located as follows:

  • 1. Styling of content re-used across multiple U-LEAF portlets should be located in the shared-ui project alongside the javascript libraries (in particular jQuery) providing the re-usable elements. Styles specified here may well override styles used for the same elements in the portal container (which also employs a version of jQuery).
  • 2. Styling of the content of a particular portlet is driven by the UI component supporting the portlet. It should re-use style information from the shared-ui project where possible, but may override and extend the style information if necessary.

    • A portlet JSP page will typically have the following style references: 

    <link rel="stylesheet" type="text/css" media="screen" href="${sharedContextPath}/css/smoothness/jquery-ui-<version>.custom.css" />
<link  rel="stylesheet" type="text/css" media="screen" href="${myContextPath}/css/<component>.css" />

Typical example could be find in SoftEntity-ui/**/jsp/searchableContainer.jsp

7. Javascript

JavaScript functions that directly reference document elements should be avoided. An element to be acted on should be passed as a parameter.

Where possible, JavaScript functions should be generalised and reusable and defined in '.js' files.

JavaScript functions that are specific to a single HTML/JSP document should be defined in the HTML/JSP document. Do not put them in shared script files. A better alternative is to rewrite them to be more generalised and reusable or at least factor out the re-usable parts.

For more information refer the section Presentation Guidelines in JSPWIKI

8. jQuery

Due to conflicts with the Liferay version of JQuery, we often create and use our own non-standard JQuery global variable (e.g. h2JQuery). There is a technique, described below, that can be used to minimise the impact of using the non-standard global. This approach should always be used in our code.

Where possible, JavaScript functions should be generalised and reusable and defined in '.js' files.

JavaScript functions that are specific to a single HTML/JSP document should be defined in the HTML/JSP document. Do not put them in shared script files. A better alternative is to rewrite them to be more generalised and reusable or at least factor out the re-usable parts.

When the jquery.ready function invokes its parameter function, it passes itself as the first parameter. By convention, we can bind this parameter to the dollar($) local name (which is the standard 'global name' for jquery). All code within the ready function can then refer to jquery as $ rather than have to use the global name. The advantage of this is that the non-standard named (e.g. h2jquery) is referenced once so when we come to change it we have much less code to modify. In addition the code is protected from any overwriting of the global variable.

Current version of Jquery used by ULEAF is v3.2.1

For more information refer the section jQuery Version Handling in JSPWIKI
